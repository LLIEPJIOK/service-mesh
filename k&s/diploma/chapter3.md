# ГЛАВА 3. ПРОЕКТИРОВАНИЕ SERVICE MESH

## 3.1 Функциональные требования

Service mesh в рамках данной работы рассматривается как инфраструктурный слой, который стандартизирует сетевое взаимодействие микросервисов в Kubernetes без внесения изменений в исходный код приложений. Соответственно, требования формулируются не только к «прокси», но и к способу внедрения (инъекции), к управлению жизненным циклом, к модели доверия и к наблюдаемости.

Ниже приведён набор ключевых функциональных требований, которые были положены в основу проектирования и реализации.

1. **Развёртывание в Kubernetes и изоляция системных компонентов.** Решение должно устанавливаться в кластер Kubernetes и размещать управляющие компоненты в отдельном системном namespace (по умолчанию `mesh-system`). Это обеспечивает изоляцию прав доступа, предсказуемость эксплуатации и отсутствие конфликтов с пользовательскими workload.

2. **Детерминированная установка и удаление.** Установка должна выполняться в фиксированном порядке, учитывающем зависимости между ресурсами (namespace → ресурсы доверия → компоненты выдачи сертификатов → конфигурация sidecar → webhook-инжекция). Удаление должно выполняться в обратном безопасном порядке и быть идемпотентным (повторный запуск не приводит к ошибкам из‑за отсутствующих ресурсов).

3. **Единый конфигурационный контракт.** Настройки mesh должны описываться единым YAML‑объектом (формат `MeshConfig`), включающим: параметры сертификатов (root CA), параметры sidecar (порты, режим mTLS, monitoring, алгоритм балансировки, retry/timeout/circuit breaker), а также политику инъекции.

4. **Автоматическая инъекция data plane.** Для целевых приложений должен применяться механизм автоматической инъекции:

- инъекция включается на уровне namespace по метке `mesh-injection: enabled`;
- инъекция может быть отключена для конкретного pod аннотацией `sidecar.mesh.io/inject: "false"`;
- в pod добавляются init‑контейнер `iptables-init` и контейнер `sidecar`, а также необходимые тома и аннотации для мониторинга.

5. **Прозрачный перехват трафика.** Входящий и исходящий TCP‑трафик приложения должен перехватываться на уровне сетевого namespace pod и перенаправляться на sidecar без изменения приложения. Для этого используются три профиля listener’ов:

- `inboundPlainPort = 15006` — входящий трафик от клиентов без mTLS (например, из ingress);
- `outboundPort = 15002` — исходящий трафик приложения к другим сервисам;
- `inboundMTLSPort = 15001` — входящий трафик от других sidecar по mTLS.

6. **Сетевая безопасность и идентичность.** Межсервисное взаимодействие внутри mesh должно выполняться по mTLS. Идентичность sidecar должна быть привязана к Kubernetes‑идентичности ServiceAccount: выпуск сертификата выполняется только после проверки JWT‑токена через Kubernetes TokenReview, а поля CSR (CN/SAN) не рассматриваются как источник истины.

7. **Service discovery и актуальный кэш endpoint’ов.** Sidecar должен получать список доступных endpoint’ов сервисов через Kubernetes API (EndpointSlice + Service) по модели LIST/WATCH, фильтруя только готовые (`Ready == true`) endpoint’ы и поддерживая потокобезопасный кэш.

8. **Балансировка нагрузки на исходящих соединениях.** Для распределения исходящих соединений по endpoint’ам должны поддерживаться как минимум два алгоритма: `roundRobin` и `random`. Алгоритм выбирается конфигурацией mesh.

9. **Базовые механизмы надежности.** На уровне исходящих соединений (на этапе dial/TLS‑handshake) должны применяться политики:

- retry с управляемым backoff (например, экспоненциальным);
- timeout на установление соединения;
- circuit breaker, предотвращающий лавинообразные попытки при деградации endpoint’ов.

10. **Observability и интеграция со стандартным стеком мониторинга.** Sidecar должен экспортировать метрики Prometheus на `/metrics` и выделенном порту `metricsPort = 9090`. Должен быть определён минимальный контракт метрик, в частности: `mesh_requests_total`, `mesh_request_duration_seconds`, `mesh_request_errors_total`, `mesh_retry_attempts_total`, `mesh_circuit_breaker_state`, `mesh_endpoints_ready`. Инъектор должен обеспечивать автоматическое обнаружение метрик через pod‑аннотации `prometheus.io/*`.

Вывод. Сформулированные функциональные требования охватывают полный жизненный цикл service mesh: от управляемой установки и конфигурации до прозрачного перехвата трафика, безопасности, надежности и наблюдаемости. Такой набор требований обеспечивает завершённость решения и пригодность для демонстрации и практического применения в Kubernetes‑окружении.

## 3.2 Архитектура решения

Архитектура service mesh построена на классическом разделении на **control plane** и **data plane**. Такое разделение позволяет с одной стороны централизованно управлять конфигурацией, безопасностью и внедрением, а с другой — выполнять обработку трафика локально в каждом pod с минимальной задержкой.

### Control plane

Control plane разворачивается в namespace `mesh-system` и состоит из следующих логических компонентов:

1. **Installer (Mesh CLI).** Командная утилита управляет установкой/удалением компонентов, применяет ресурсы в фиксированном порядке и при необходимости ожидает готовности критичных сервисов. Конфигурация задаётся через `MeshConfig`.

2. **Admission webhook (hook).** Компонент, который принимает admission‑запросы Kubernetes для объектов `pods` и выполняет строго ограниченную мутацию pod template: добавляет `iptables-init`, `sidecar`, volume с корневым CA и служебные аннотации. Включение инъекции производится через `namespaceSelector` по метке `mesh-injection: enabled`, а выключение — через аннотацию pod `sidecar.mesh.io/inject: "false"`. Политика отказа webhook устанавливается как `failurePolicy: Fail`, что гарантирует недопущение неконсистентных workload без sidecar в целевых namespace.

3. **Менеджер сертификатов (cert-manager).** Сервис, который обеспечивает выпуск leaf‑сертификатов для sidecar на основе identity из ServiceAccount токена. Перед выпуском сертификата выполняется проверка токена через TokenReview, а также проверка подписи CSR (`csr.CheckSignature()`). Root CA хранится в системных ресурсах mesh и приватный ключ не монтируется в pod’ы приложений.

4. **Ресурсы доверия и конфигурации.** Control plane опирается на Kubernetes ресурсы:

- Secret `mesh-root-ca` — хранение корневого сертификата и ключа;
- Secret `mesh-webhook-tls` — TLS‑сертификат webhook‑сервера;
- ConfigMap `mesh-sidecar-config` — канонические значения конфигурации sidecar, используемые при инъекции;
- RBAC: роль для `TokenReview` (для cert-manager) и роль для service discovery (для workload/sidecar в namespace приложений).

### Data plane

Data plane располагается рядом с приложением и состоит из двух частей:

1. **Init‑контейнер `iptables-init`.** Запускается первым, получает права `NET_ADMIN`, настраивает правила `iptables` в таблице `nat` и завершает работу. Входящий трафик перенаправляется через `PREROUTING`, исходящий — через `OUTPUT`. В правилах предусмотрены исключения (например, `metricsPort = 9090` не перехватывается), а также предотвращение петель за счёт исключения трафика пользователя sidecar по UID `1337`.

2. **Sidecar‑прокси.** Контейнер с прокси‑функциями и middleware‑цепочкой обработки соединений. Sidecar принимает соединения на портах `15006/15002/15001`, восстанавливает исходное назначение соединения через `SO_ORIGINAL_DST`, выполняет service discovery, выбирает endpoint (balancing), при необходимости устанавливает mTLS‑соединение, применяет retry/timeout/circuit breaker, а также экспортирует метрики на `/metrics:9090`.

### Поток обработки при создании pod

Последовательность запуска в mesh (уровень pod):

1. Kubernetes получает запрос на создание/обновление pod.
2. Admission webhook выполняет мутацию, добавляя init‑контейнер и sidecar, необходимые volume и аннотации.
3. Pod запускается, `iptables-init` применяет правила и завершается.
4. Sidecar стартует параллельно с приложением, поднимает listener’ы и выполняет TLS bootstrap (получение сертификата у cert-manager).
5. Приложение работает в штатном режиме; сетевой трафик прозрачно проходит через sidecar.

Вывод. Архитектура решения обеспечивает чёткую границу ответственности: control plane управляет установкой, инъекцией и доверием, а data plane реализует обработку трафика и наблюдаемость «на месте». Это повышает управляемость и воспроизводимость развертываний, а также упрощает расширение mesh на новые namespace и workload.

## 3.3 Выбор технологического стека и инструментов

Выбор стека был ориентирован на три критерия: (1) максимальная интеграция со стандартными механизмами Kubernetes, (2) высокая эффективность и простота сопровождения сетевых компонентов, (3) совместимость с распространённым observability‑стеком.

1. **Go как основной язык реализации.** Компоненты control plane и data plane реализуются на Go. Язык выбран из‑за зрелых сетевых библиотек, удобной конкурентной модели (goroutine), встроенной поддержки TLS/X.509, а также хорошей совместимости с контейнерными окружениями.

2. **Kubernetes API как источник истины.** В решении используются:

- Admission registration API (MutatingWebhookConfiguration) для автоматической инъекции;
- Authentication API (TokenReview) для подтверждения ServiceAccount токенов;
- Discovery API (EndpointSlice) и Core API (Service) для service discovery;
- RBAC для ограничения прав и описания обязательных ролей.

3. **Linux сетевые примитивы.** Для прозрачности перехвата используется `iptables` (таблица `nat`, цели `REDIRECT`, цепочки `PREROUTING` и `OUTPUT`). Для восстановления оригинального назначения после `REDIRECT` применяется `SO_ORIGINAL_DST`, что позволяет sidecar маршрутизировать соединение по исходному `ClusterIP:port`.

4. **Формат конфигурации.** Для оператора выбран YAML (`MeshConfig`) как наиболее распространённый формат декларативных настроек в Kubernetes‑экосистеме. Для передачи параметров в контейнеры используется набор переменных окружения, инжектируемых webhook’ом.

5. **Наблюдаемость.** Для мониторинга используется Prometheus‑совместимый формат метрик и Grafana для визуализации. Развёртывание мониторинга производится через Helm‑чарт `kube-prometheus-stack`, что обеспечивает повторяемость и согласованность с индустриальными практиками.

6. **Среда демонстрации и валидации.** Для воспроизводимой проверки используется minikube, ingress‑контроллер NGINX для доступа к фронтенд‑странице, а также эталонное микросервисное приложение BookInfo, удобное для демонстрации распределения запросов по версиям сервиса.

Вывод. Подбор инструментов обеспечивает баланс между практической ценностью и инженерной корректностью: Kubernetes остаётся источником истины для идентичности и topology, Go даёт эффективную реализацию сетевого proxy, а Prometheus/Grafana делают качество работы mesh измеримым и проверяемым.

## 3.4 Реализация control plane

Реализация control plane в разработанном решении строится вокруг идеи «контракт‑первым»: оператор описывает систему одним конфигурационным файлом `MeshConfig`, а далее установка и поддержание целостности выполняются автоматически.

### MeshConfig как единый контракт установки

`MeshConfig` содержит:

- `spec.namespace` и `spec.version` — системный контур и версия;
- `spec.images.*` — закрепление образов sidecar/iptables-init/cert-manager;
- `spec.certificates.rootCA` — root CA (сертификат и ключ) как основа доверия;
- `spec.sidecar.*` — порты, режим mTLS, monitoring, алгоритм балансировки, политики reliability;
- `spec.injection.namespaceSelector` — правило включения инъекции по меткам.

Такой подход позволяет не разносить параметры по множеству разрозненных файлов и снижает вероятность несогласованной конфигурации.

### Детерминированная установка через Mesh CLI

CLI‑утилита предоставляет основной сценарий: `mesh install -f <config.yaml>`. При установке соблюдается фиксированный порядок действий:

1. Создание namespace `mesh-system` (или заданного в конфигурации).
2. Размещение root CA в Secret `mesh-root-ca`.
3. Развёртывание cert-manager вместе с необходимым RBAC для TokenReview.
4. Размещение ConfigMap `mesh-sidecar-config` с параметрами sidecar по умолчанию.
5. Развёртывание webhook‑сервера и подготовка TLS‑секрета `mesh-webhook-tls`.
6. Применение `MutatingWebhookConfiguration` с `failurePolicy: Fail` и `namespaceSelector`.

Важным свойством является режим ожидания готовности (`--wait` с `--timeout`): установка считается успешной только после readiness критичных компонентов. Для эксплуатации это означает, что оператор получает понятное подтверждение корректного развертывания без «скрытых» ошибок.

### Webhook-инжекция и безопасность pod template

Webhook реализует строгий контракт мутации и не затрагивает поля, не указанные в контракте. Это важно по двум причинам:

1. **Предсказуемость.** Инъекция не ломает пользовательские настройки (например, не подменяет уже заданный `serviceAccountName`).

2. **Безопасность.** Компонент работает в системном namespace, использует TLS (контейнер слушает `:8443`, сервис публикует `443`), имеет readiness/liveness‑пробы и усиленный securityContext (например, `readOnlyRootFilesystem`, запрет `allowPrivilegeEscalation`, `seccompProfile: RuntimeDefault`).

Webhook добавляет:

- init‑контейнер `iptables-init` с `NET_ADMIN` capability и параметрами портов/исключений;
- контейнер `sidecar` с портами `15001/15002/15006`, uid `1337` и монтированием root CA;
- volume `mesh-ca` (ConfigMap с корневым сертификатом);
- аннотации Prometheus scrape (`prometheus.io/scrape`, `prometheus.io/port`, `prometheus.io/path`) при включённом monitoring.

### Менеджер сертификатов и модель доверия

Менеджер сертификатов реализует HTTP API `POST /sign` и выдаёт leaf‑сертификаты для sidecar. Логика выпуска включает:

1. Получение JWT‑токена ServiceAccount из pod.
2. Валидацию токена через Kubernetes TokenReview (требуется ClusterRole с правом `create` на `tokenreviews`).
3. Проверку корректности CSR и подписи CSR.
4. Формирование identity сертификата из валидированного токена (namespace + ServiceAccount), а не из полей CSR.
5. Подпись leaf‑сертификата root CA, находящимся в защищённом Secret.

Для эксплуатации предусмотрены типовые механизмы устойчивости сервиса: ограничение максимального размера запроса, timeouts на HTTP‑сервер, graceful shutdown и возможность включения rate limit.

Вывод. Control plane реализован как законченный контур управления: единая конфигурация, детерминированная установка, строгий контракт инъекции и проверяемая модель доверия для mTLS. Это обеспечивает воспроизводимый запуск mesh и снижает риск неконсистентного состояния приложений в namespace, где включена инъекция.

## 3.5 Реализация sidecar-прокси

Sidecar представляет собой прокси‑контейнер, внедряемый рядом с приложением, и является основной реализацией data plane. Проектирование sidecar начинается с требования прозрачности: приложение должно продолжать работать с теми же адресами (ClusterIP сервисов, ingress), но фактически весь сетевой трафик должен проходить через прокси.

### Перехват трафика: iptables-init

Init‑контейнер настраивает `iptables` в таблице `nat` и обеспечивает:

- inbound‑перехват: `PREROUTING → MESH_INBOUND → REDIRECT` на `15006` для портов приложения;
- outbound‑перехват: `OUTPUT → MESH_OUTPUT → REDIRECT` на `15002` для TCP‑трафика приложения;
- исключение трафика sidecar по UID `1337` для предотвращения петель;
- исключение DNS (порт 53), а также исключения по списку портов и IP/CIDR (например, исключение `9090` для метрик и `169.254.169.254/32` как типового metadata endpoint).

С точки зрения эксплуатации важно, что скрипт iptables реализован идемпотентно: повторный запуск не создаёт дубликаты цепочек и правил.

### Listener-профили и маршрутизация

Sidecar поднимает три listener’а:

- `15006` — для трафика, перенаправленного iptables с inbound‑порта приложения (plain);
- `15002` — для всего исходящего трафика приложения, перенаправленного iptables;
- `15001` — для входящих соединений от других sidecar, которые уже используют mTLS.

Для соединений, перенаправленных через `REDIRECT`, sidecar восстанавливает оригинальное назначение через `SO_ORIGINAL_DST`. Это ключевой элемент, позволяющий определить, к какому сервису обращалось приложение на самом деле (обычно это `ClusterIP:port`).

### Service discovery на основе EndpointSlice

После получения `OriginalDst` sidecar должен сопоставить `ClusterIP:port` с логическим именем сервиса и набором endpoint’ов. Для этого используется связка:

1. LIST/WATCH `Service` — построение и поддержание маппинга `serviceName:port → clusterIP:port`.

2. LIST/WATCH `EndpointSlice` — получение актуального набора endpoint’ов, сохранение в кэше по двум ключам: `serviceName:port` и `clusterIP:port`. В кэш попадают только endpoint’ы `Ready == true`.

Кэш реализуется потокобезопасно, чтобы не блокировать обработку трафика при обновлениях watch‑потоков.

### Балансировка нагрузки

Получив список endpoint’ов, sidecar выбирает целевой endpoint согласно выбранному алгоритму:

- `roundRobin` — поочерёдный выбор endpoint’ов;
- `random` — случайный выбор endpoint’а.

Алгоритм задаётся централизованно конфигурацией mesh и, таким образом, одинаков для всего mesh‑контура.

### mTLS и проверка доверия

При установлении соединения sidecar определяет, является ли адрес «внутримешевым»: если endpoint найден в discovery‑кэше, соединение считается mesh‑внутренним и устанавливается по mTLS. Для mTLS используются leaf‑сертификаты, выпущенные cert-manager, а проверка peer‑сертификата выполняется по корневому CA.

Трафик к адресам, не найденным в discovery‑кэше (внешние зависимости), передаётся обычным TCP‑соединением без mTLS, что сохраняет совместимость с внешними сервисами.

### Надежность: retry/timeout/circuit breaker

Поскольку sidecar является единой точкой для сетевых операций, в нём реализованы базовые механизмы надежности на этапе установления исходящего соединения:

- **Retry** применяется для ошибок dial/TLS‑handshake/timeout и не затрагивает прикладной протокол (например, HTTP‑коды). Это уменьшает вероятность случайных сетевых отказов.
- **Timeout** ограничивает время установления соединения и предотвращает зависания.
- **Circuit breaker** переводит endpoint в состояние «open» после превышения порога ошибок и на время блокирует попытки, давая сервису восстановиться.

Такое поведение повышает устойчивость межсервисной коммуникации и делает деградации измеримыми на уровне метрик.

### Observability: метрики sidecar

Sidecar экспортирует метрики Prometheus на `/metrics` и `metricsPort = 9090`. Минимальный контракт включает:

- `mesh_requests_total` — количество проксируемых запросов/соединений;
- `mesh_request_duration_seconds` — распределение латентности;
- `mesh_request_errors_total` — сетевые/прокси ошибки;
- `mesh_retry_attempts_total` — количество повторных попыток;
- `mesh_circuit_breaker_state` — состояние circuit breaker;
- `mesh_endpoints_ready` — количество готовых endpoint’ов.

Метрики используются как для валидации правильности прохождения трафика через sidecar, так и для оперативной диагностики проблем.

Вывод. Реализация sidecar-прокси обеспечивает полный набор функций data plane: прозрачный перехват трафика, маршрутизацию на основании исходного назначения, динамическое обнаружение endpoint’ов, балансировку, mTLS и механизмы надежности. Наблюдаемость через метрики делает работу mesh проверяемой и управляемой в эксплуатации.

## 3.6 Реализация CLI утилиты

CLI‑утилита рассматривается как интерфейс оператора к системе: она должна не только запускать установку, но и обеспечивать повторяемость сценариев развёртывания приложений и мониторинга. Для согласования со структурой курсовой работы выделим три функциональных блока: `plane`, `deploy`, `prom`.

### 3.6.1 Функция plane

Функция `plane` отвечает за развертывание и жизненный цикл управляющего контура (control plane) и базовой конфигурации mesh.

Содержательно она реализуется командами:

- `mesh install -f <mesh-config.yaml>` — установка: валидация `MeshConfig`, применение ресурсов в фиксированном порядке, ожидание готовности критичных компонентов (например, webhook и cert-manager) с таймаутом.
- `mesh uninstall` — удаление: удаление ресурсов в обратном порядке с идемпотентностью (отсутствующий ресурс не приводит к аварийному завершению).

Кроме базового сценария предусмотрены эксплуатационные возможности:

- выбор namespace через `-n/--namespace`;
- управление ожиданием готовности и таймаутом (`--wait`, `--timeout`);
- вывод ресурсов без изменения кластера (`--dry-run`).

Функция `plane` обеспечивает инженерно важное свойство — **детерминированность**. При одинаковом `MeshConfig` оператор получает одинаковый набор ресурсов в одинаковом порядке, что важно для воспроизводимости и проведения демонстраций.

Вывод. Функция `plane` формирует основу управляемого service mesh: она гарантирует правильную последовательность установки, валидирует конфигурацию и подтверждает готовность ключевых компонентов перед передачей системы в эксплуатацию.

### 3.6.2 Функция deploy

Функция `deploy` покрывает сценарии развёртывания workload в mesh и проверки включения инъекции на реальном приложении.

В качестве эталонного workload используется BookInfo, так как он включает несколько микросервисов и несколько версий сервиса `reviews`, что позволяет демонстрировать распределение трафика. Основные действия в блоке `deploy`:

1. **Подготовка namespace приложения.** Workload разворачивается в namespace с меткой `mesh-injection: enabled`, что включает автоматическую инъекцию для всех pod внутри.

2. **ServiceAccount и RBAC для discovery.** Для предсказуемой identity и для доступа sidecar к Kubernetes API (Service и EndpointSlice) используются ServiceAccount’ы и роль `pod-reader` в namespace приложения.

3. **Развёртывание приложения.** Применяется kustomize‑набор манифестов BookInfo, создающий deployments и services для `productpage/details/ratings/reviews`.

4. **Проверка распределения трафика.** Многократные запросы к `productpage` должны демонстрировать участие разных версий `reviews`, что подтверждает корректность межсервисных вызовов и наличие балансировки на стороне data plane.

Вывод. Функция `deploy` переводит mesh из инфраструктурной установки в прикладной сценарий: она демонстрирует, что приложения подключаются к mesh декларативно (через метки/манифесты) и начинают обмениваться трафиком через sidecar без изменения кода.

### 3.6.3 Функция prom

Функция `prom` отвечает за развёртывание мониторинга и подтверждение наблюдаемости mesh.

Сценарий включает:

1. **Установка Prometheus и Grafana.** Развёртывание выполняется через Helm‑чарт `kube-prometheus-stack` с проектными значениями (values). Такой подход обеспечивает повторяемость окружения и стандартный набор CRD/операторов.

2. **Подключение сбора метрик sidecar.** В кластере создаётся PodMonitor, который скрейпит `/metrics` на порту `9090` в pod’ах BookInfo (sidecar). Дополнительно инъектор обеспечивает pod‑аннотации `prometheus.io/*`, что делает поведение прозрачным для оператора.

3. **Визуализация в Grafana.** Подготавливается dashboard, позволяющий наблюдать request count, latency, ошибки, состояние circuit breaker и готовность endpoint’ов.

Вывод. Функция `prom` завершает эксплуатационный контур: она делает работу service mesh измеримой, а диагностику — воспроизводимой. Наличие метрик и их визуализация подтверждают, что трафик проходит через data plane и что политики надежности влияют на наблюдаемые показатели.

Вывод. В рамках CLI‑утилиты выделенные функциональные блоки `plane/deploy/prom` покрывают полный сценарий работы оператора: установка control plane, подключение приложений к mesh и включение наблюдаемости. Такое разделение делает процесс развертывания понятным, повторяемым и удобным для демонстрации.

## 3.7 Тестирование

Для подтверждения работоспособности системы используется набор smoke‑сценариев, позволяющих быстро проверить три ключевых аспекта: доступность приложения для клиента, межсервисную коммуникацию и наличие метрик.

### 3.7.1 Тестирование доступности сервиса для клиента

Цель — подтвердить, что после установки mesh и развертывания приложения конечный пользователь получает доступ к пользовательской странице BookInfo.

Методика:

1. Выполняется установка control plane (функция `plane`).
2. Выполняется развёртывание BookInfo (функция `deploy`).
3. Проверяется доступ к `productpage` через ingress или fallback‑доступ через NodePort `31380`.
4. Проверка считается успешной, если страница возвращает HTTP 200 и корректное содержимое.

Пример проверки:

```bash
curl -i "http://$(minikube ip):31380/productpage" | head -n 1
```

Вывод. Тест доступности подтверждает, что внедрение service mesh не нарушает внешний пользовательский путь и что приложение остаётся доступным клиентам после включения data plane.

### 3.7.2 Тестирование доступности межсервисной коммуникации

Цель — подтвердить корректность вызовов между микросервисами внутри приложения (productpage → details/ratings/reviews) и продемонстрировать распределение запросов по нескольким версиям сервиса `reviews`.

Методика:

1. Генерируется серия последовательных запросов к `productpage`.
2. В ответах анализируются признаки версии `reviews`.
3. Успешным считается сценарий, при котором в серии ответов наблюдаются разные варианты (v1/v2/v3), а сам пользовательский поток не прерывается.

Пример проверки:

```bash
for i in $(seq 1 20); do
  curl -s "http://$(minikube ip):31380/productpage" \
    | grep -E "(Reviews served by|glyphicon-star|color=black)" \
    | head -n 1
done
```

Вывод. Проверка межсервисной коммуникации демонстрирует, что service discovery и балансировка на стороне sidecar корректно выбирают endpoint’ы, а mTLS и политики надежности не препятствуют нормальной работе бизнес‑сценариев приложения.

### 3.7.3 Просмотр метрик

Цель — подтвердить, что наблюдаемость работает на практике: метрики sidecar попадают в Prometheus и отображаются в Grafana.

Методика:

1. Развёртывается мониторинг (функция `prom`).
2. В Prometheus выполняется запрос к ключевым метрикам (например, `mesh_requests_total`).
3. В Grafana открывается dashboard и проверяется наличие ненулевых значений по request count и latency на текущем интервале.

Пример проверки Prometheus API:

```bash
curl -s "http://$(minikube ip):32001/api/v1/query?query=mesh_requests_total" | jq '.status'
```

Вывод. Наличие метрик в Prometheus и корректная визуализация в Grafana подтверждают, что трафик действительно проходит через sidecar и что эксплуатационные свойства mesh (ошибки, латентность, retries, состояние circuit breaker) доступны оператору в измеримом виде.

Вывод по главе 3. В главе спроектирована и описана архитектура service mesh для Kubernetes как завершённого решения, включающего control plane и data plane, строгую модель доверия для mTLS, автоматическую инъекцию в workload, а также наблюдаемость и проверку работоспособности через тестовые сценарии. Предложенная архитектура обеспечивает воспроизводимое развертывание и управляемую межсервисную коммуникацию для микросервисных приложений.
